12-Node LangGraph Architecture

A single typed state machine with deterministic edges, four conditional feedback loops, and explicit hard-stop conditions at every gate.

4
Agent Breakdown

Each agent owns a narrow, explicit responsibility. Static system prompts stay cache-friendly; retry feedback enters only through the human message.

Schema Agent

CSV interpretation, entity guess, primary keys, foreign keys, column semantics.

Ontology Scout

No LLM call. Deterministic RDFLib extraction of classes, prefixes, and property hierarchy.

Mapper Agent

Links each column to a real ontology URI and lists required prefixes.

Schema Alignment Agent

Builds the functional entity plan, multi-node structure, and retry-aware replanning.

CQ Generator

Auto-generates Competency Questions from schema + entity plan when no user CQs are provided.

YARRRML Coordinator

Orchestrates prefix, entity, and relationship agents plus deterministic post-processing.

Refiner Agent

Column coverage, prefix auto-fix, IRI and FK validation, constraint injection on failure.

CQ→SPARQL Agent

Translates each CQ to a SPARQL ASK query grounded in YARRRML prefixes and live KG types. Self-corrects on parse errors.

SHACL Validator

Derives shapes from the ontology via Astrea; runs pyshacl against the KG; returns structured violation feedback.

Why this decomposition works

  • Shorter prompts → lower hallucination pressure.
  • Clearer failure attribution per node.
  • Better retry targeting on specific errors.

What's new vs. previous version

  • CQ Generator auto-creates validation targets.
  • CQ→SPARQL agent executes checks on the actual KG.
  • SHACL node is dataset-agnostic (ontology-derived shapes).
5
Competency Question Pipeline

CQs serve two roles: they guide YARRRML generation toward the intended semantic structure, then act as executable validation targets on the materialized KG.

In
CQ Source

User-provided via --cqs flag, or auto-generated from schema + entity plan.

1
CQ Generator Agent

Produces 5–10 questions covering entity existence, attributes, and relationships. Grounded to actual entity types in the plan.

2
Generation guidance

CQs injected into YARRRML coordinator prompt to bias entity and predicate choices.

3
CQ → SPARQL Agent

Each CQ translated to an ASK query, grounded in YARRRML prefixes and live KG probe results. Self-corrects on parse errors.

4
SPARQL Execution

ASK queries run against the materialized KG via pyoxigraph. Failures trigger structured feedback with exact triple patterns.

Three input modes

  • --cqs "..." — user natural-language CQs (translated by LLM)
  • --sparql-queries "ASK {...}" — direct SPARQL (no LLM)
  • Auto-generated — triggered when no CQs are provided

KG-grounded translation

Before translating CQs, the agent probes the live KG for actual classes and predicates. Prevents writing ?s a schema:Order when only schema:OrderItem exists.

Retry routing

  • Retries 0–1 → regenerate YARRRML with targeted fix
  • Retries ≥ 2 → escalate to schema re-alignment
  • Cap at CQ_SPARQL_MAX_RETRIES

Diagnosis feedback

On failure, the exact failing triple patterns and SELECT probe diagnostics are injected into the next generation prompt.

# Example generated CQ
"Which transactions are linked to a merchant?"

# Resulting SPARQL ASK
PREFIX ex: <http://example.org/>
ASK {
  ?t a ex:CreditCardTransaction ;
     ex:atMerchant ?m .
  ?m a ex:Merchant .
}
6
4-Layer Self-Correction Loop

Errors are intercepted from cheapest to most expensive. Each layer feeds back precise, actionable constraints rather than generic criticism.

Layer 1 — Syntax

  • Yatter translation is deterministic.
  • Broken YARRRML + traceback injected into next prompt.
  • Auto-repair tries common fixes before full regeneration.
  • Hard stop at 10 retries.

Layer 2 — Logic

  • Deterministic checks first: column coverage, IRI flags, FK patterns, prefix declarations.
  • LLM critique only for ambiguous cases.
  • Non-actionable feedback filtered out.
  • Hard stop at 6 retries.

Layer 3 — SHACL

  • Astrea derives shapes from ontology automatically.
  • pyshacl validates materialized KG.
  • Deduplicated violation summaries fed back.
  • Pass-through when --shacl not set.

Layer 4 — SPARQL CQ

  • CQs executed as ASK queries on live KG.
  • Exact failing triple patterns in feedback.
  • Deep failures escalate to schema re-alignment.
  • Cap via CQ_SPARQL_MAX_RETRIES.
stateDiagram-v2 [*] --> Generate Generate --> Syntax Syntax --> Generate: SYNTAX_ERROR Syntax --> Logic: PASSED Logic --> Generate: LOGIC_ERROR Logic --> KG: APPROVED KG --> SHACL SHACL --> Generate: SHACL_ERROR SHACL --> SPARQL: pass/skip SPARQL --> Generate: CQ fail SPARQL --> Align: deep fail Align --> Generate SPARQL --> [*]: pass/skip

Combined retry budget

All four loops share a single retry_count for YARRRML regeneration, with independent counters for SPARQL CQ retries. Later retries carry increasingly precise column-level and triple-pattern constraints.

7
Schema Alignment

Instead of dumping all columns into one mapping, the alignment agent produces a functional entity plan — a multi-node blueprint that drives every subsequent agent.

❌ Flat Anti-Pattern

All columns collapsed into one mapping. Merchants are plain strings, person details are literals, no object-property links exist. CQs like "which merchant processed a fraud transaction?" become impossible to answer.

TransactionMapping:
  s: ex:Transaction/$(trans_num)
  po:
    - [ex:merchant,  $(merchant), xsd:string]  # string, not IRI
    - [ex:firstName, $(first),    xsd:string]
    # No entity links, no address node, no metadata

✅ Functional Entity Plan

The alignment agent detected 5 semantic nodes, IRI-valued columns, inverse links, and a metadata sub-entity — producing this blueprint before any YARRRML is written.

1. TransactionMapping   → ex:CreditCardTransaction
   links: CreditCard~iri, Merchant~iri, Metadata~iri

2. MerchantMapping      → ex:Merchant
   inverse: eli:has_part → Transaction~iri

3. PersonMapping        → schema:Person
   link → Address~iri (schema:address)

4. AddressMapping       → schema:PostalAddress

5. TransactionMetadataMapping → ex:CreditCardTransactionMetadata

🔗 Metadata node splitting

Ontology property lkg:metadata detected → separate metadata mapping created with URI ex:Transaction/$(trans_num)/Metadata.

👆 Inverse parent links

Merchant and Person mappings carry eli:has_part → Transaction~iri so you can navigate from any entity back to its transactions.

🌐 IRI column recognition

cc_num and merchant recognized as identifier columns — mapped as ~iri inline, not as string literals.

8
YARRRML Coordinator

YARRRML production is split into focused sub-agents and a deterministic post-processing chain, avoiding one large hallucination-prone generation step.

In
Functional Entity Plan + CQs

Produced by schema alignment and CQ generator.

1
PrefixAgent

Generates and caches the prefix block. Skipped on retry when no new namespaces appear.

2
EntityAgent

Builds mappings, subjects, rdf:type entries, and data property assertions.

3
RelationshipAgent

Adds only missing object-property links when multi_node=true. Avoids re-emitting the full entity block.

Out
Final YARRRML

Normalized paths, reconciled prefixes, debug artifacts saved per attempt.

Prefix cache strategy

Prefix regeneration skipped when the entity plan introduces no new namespaces. Cache invalidated on alignment re-runs.

Auto-prefix reconciliation

Undeclared prefixes resolved from ontology, a well-known prefix table, or a fallback namespace — without consuming a retry.

Token efficiency

Relationship agent avoids re-emitting the whole entity block, saving ~12K tokens per retry.

Debug trail

Each attempt saves prefixes, entities, intermediate, and after_links YAML artifacts.

# Coordinator steps
# 1. prefixes = cached or regenerated
# 2. entities = entity agent output
# 3. intermediate = prefixes + mappings
# 4. optional relationship enrichment
# 5. normalize CSV source paths
# 6. reconcile missing prefixes
9
Refiner Agent

The logic gate runs deterministic checks first, then uses LLM critique only where automation cannot safely decide.

Column coverage

Extracts all referenced $(columns), compares against CSV headers, and ignores known-empty columns via cached checks.

Prefix auto-fix

Resolves missing declarations from ontology prefixes or a known-prefix table without consuming a retry slot.

IRI validation

Ensures URL-valued columns use ~iri rather than literal mapping treatment.

Syntax normalization

Accepts and normalizes brace-style templates and composite key patterns before the LLM sees them.

Constraint injection

On logic failure, the next retry receives per-column predicate and mapping constraints from the refiner.

Why retries reduce over time

The refiner feeds back actionable, mapping-specific constraints rather than generic criticism, narrowing the search space for each generation attempt.

Why false positives matter

Non-actionable LLM feedback is filtered so harmless observations do not create endless logic-error loops.

Evaluation Framework

Three metric levels let pipeline health, KG fidelity, and structural completeness be measured independently and combined.

Level 1 — Pipeline health

Always available, zero cost.

yarrrml producedsyntax validtranslatablematerializablesuccessretry counttripleslatency

Level 2 — Gold comparison

Precision, recall, and F1 after URI-tolerant normalization across triple, predicate, and class dimensions.

triple F1predicate F1class F1object type mismatch

Level 3 — Column coverage

Structural completeness through YARRRML template analysis and literal-value matching.

columns totalYARRRML coveragevalue coveragemissing columns

How to run

python main.py --eval 1 2 3 --gold data/gold/bikeshare_gold.nt
python main.py --eval 1 --sparql --cqs "Which stations have docks?"
python main.py --eval 1 --shacl --sparql

SPARQL CQ results are saved alongside eval_metrics.json in the run directory.

URI normalization for fair L2 comparison

  • Subject URIs collapse to row IDs.
  • Object URIs collapse to local names or referenced IDs.
  • Literals compare by lexical value.
  • IRI vs literal type preserved for diagnostics.

LLM Configuration & CLI

A flexible multi-provider system supports role-specific models, temperatures, and structured output mode. New flags unlock optional validation layers.

LM Studio — local default

LLM_PROVIDER=lm_studio
LM_STUDIO_URL=http://localhost:1234/v1
LLM_MODEL_DEFAULT=qwen2.5-coder-14b-instruct
  • Full privacy and offline capability
  • Any OpenAI-format local server works
  • KV-cache helps with static system prompts

Comet AI — cloud option

LLM_PROVIDER=comet
COMET_API_KEY=your_key_here
COMET_MODEL=gpt-4o
  • Access to frontier cloud models
  • Same ChatOpenAI-compatible client path
  • Global model override or per-role defaults
